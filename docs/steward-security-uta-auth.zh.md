# Steward 安全议题 — UTA 交易接口的信任边界（讨论稿）

> 版本：草案 v0.1（2026-07-06）
> 地位：**讨论稿**，不是执行计划。P3-4a 活体审查（安全权重最高的一次）抛出的两个 CRITICAL 触及项目根基——"笼子可信"到底成不成立。本文把问题讲清楚，maintainer 批注收敛后再决定怎么改、排在哪。
> 用法：maintainer 在 §5 的问题下用 `>` 批注，orchestrator 内联答复升版本。**P3-4a 代码已实现但暂不提交，等本文收敛。**

---

## 1. 一句话版本

**一个跑在工作区里的 AI agent，可以绕过我们所有的人工审批设计，直接命令 UTA 下单成交。**

它不需要"push 工具"（我们 P3-1 已经把 push 工具删了），它只要在自己的终端里执行一条网络请求就行。我们所有"人类专属 push""无 push 工具"的保证，挡不住这条路。

---

## 2. 先复习：我们以为的架构长什么样

回顾一下我们一路建立的安全模型（P1/P2/P3）：

```
                     ┌─────────────────────────────────────┐
   AI agent 在       │  Alice 进程                          │
   workspace 里  ──> │  - 给 agent 的工具面（P3-1 裁剪过）  │
   （通过 MCP 工具） │  - 人类专属 push 路由（要登录会话）  │
                     │  - 审批门、审计                       │
                     └───────────────┬─────────────────────┘
                                     │  HTTP 转发
                                     ▼
                     ┌─────────────────────────────────────┐
                     │  UTA 进程（管 broker、真正下单）    │
                     │  - guards 管道、风险状态机           │
                     │  - 只有它能碰 broker 凭证            │
                     └─────────────────────────────────────┘
```

我们一直假设的因果链是：

- agent 只能通过 **MCP 工具**说话（P3-1 裁剪了工具面，删了 push 工具）；
- 要真正成交，必须有人在 **Alice 的 Web UI** 点 push（人类专属路由，要登录会话）；
- Alice 再把请求转发给 UTA，UTA 才碰 broker。

**结论（我们以为的）**：agent 没有 push 工具 ⟹ agent 不能让交易成交 ⟹ 必须人点头。

---

## 3. 实际上的架构——洞在哪

问题出在 **Alice 和 UTA 之间那条线**，以及**工作区 agent 其实站在哪里**。

### 3.1 UTA 对谁都不设防

UTA 的交易接口（包括"人类专属"的 wallet/push、emergency-stop、flatten）**完全不检查请求者是谁**。代码里的原话注释（`src/webui/routes/trading-proxy.ts:6-7`）：

> "has no auth between the two — UTA is bound to 127.0.0.1 only, so the trust boundary is the host, not the request."
> （两者之间没有认证——UTA 只绑在 127.0.0.1，所以信任边界是"这台机器"，不是"这个请求"。）

翻译成人话：**UTA 的设计假设是"能连到我这个本机端口的，都是自己人，照做不误"。** 它不看你是不是登录过、是不是 Alice、是不是人——只要请求打到 `127.0.0.1:<UTA端口>`，它就执行。

这个设计本身在"Alice 和 UTA 都是我们自己的进程"的世界里是合理的。问题是——

### 3.2 工作区 agent 也站在这台机器上

工作区里的 AI agent 不是在一个隔离的盒子里。它是**同一台机器上的一个普通进程**（PTY 终端），它能：

- 访问 `127.0.0.1`（loopback，本机网络，没有任何东西拦它）；
- 读到 UTA 的默认端口号 **47333**——这个号写在 **CLAUDE.md** 里，而 CLAUDE.md 就在 agent 能读到的仓库里（退一步，就算不写，扫一下本机端口也是几秒钟的事）；
- 执行任意 shell 命令（`shell` 适配器直接就是终端；`codex`/`claude` 也能跑命令）。

我们确实在 P3-1/#32 做了一层防护：把 Alice 的工具 URL、MCP URL、内部 token 都从 agent 的环境变量里**剥掉了**。所以 agent 的环境里看不到这些地址。但这挡不住它**知道默认端口 + 直接连 loopback**。

### 3.3 于是这条路是通的

```
AI agent 在 workspace 终端里，直接执行：

  curl -X POST http://127.0.0.1:47333/uta/<账户id>/wallet/push

  （账户 id 它从自己的只读查询工具就能拿到）

UTA 收到 → "来自本机，照做" → 执行待批交易 → 成交。

全程：没有 push 工具、没有登录会话、没有人点头。
```

活体审查已经**实测复现**了这条路（审查员用裸 curl 对一个 live 类型账户打 wallet/push，唯一失败点是我们给的假 broker 凭证被 broker 拒了——也就是说，授权这一关它**根本没被拦**，只是假凭证下不了真单）。

### 3.4 P3-4a 让问题更具体（F1）

P3-4a（paper auto-push）新加的一个点，是同一类问题的新实例：它从 HTTP 请求体里的 `effectiveAuthzLevel` 字段判断"这个 workspace 够不够 paper 档、能不能自动放行"。这个字段可以伪造——裸 curl 灌一个 `effectiveAuthzLevel: "paper"`，auto-push 就开火了。UTA 只用账户侧的上限二次钳制，但 workspace 这一侧的档位是**照单全收 HTTP 请求里写的值**。

好在 P3-4a 的**实盘闸是结构性的、扛住了**（审查 F3：对 live 账户，无论怎么伪造都不会 auto-push），所以 F1 今天的实际杀伤只在 paper（虚拟钱）。但它和 §3.3 的洞是同一类。

## 4. 为什么这件事严重到要停下来讨论

我们整个项目的命门，你定的那句话——**"bot 不可信，笼子可信"**。

这个发现说的是：**笼子的锁，本质是"agent 不知道钥匙在哪"，而不是"agent 拿不到钥匙"。** 一旦 agent 知道了（而钥匙就写在它能读的 CLAUDE.md 里），锁就是虚的。

我们在 P1（风险脊柱）、P2（审计链）、P3（授权阶梯）建的每一道人工审批、每一层授权，都默认"agent 只能走 Alice 给的正门"。这个发现说正门旁边有一扇没锁的后门。**不是这些工作白做了**——guards、风险状态机、审计在正门上全部有效；但只要后门开着，一个"想绕"的 agent 可以不走正门。

诚实地说：这个洞是**早就存在的**（人类专属 push 一直建在这个不设防的 UTA 接口上），不是 P3-4a 引入的。P3-4a 只是第一个"让 agent 自主执行"的功能，把这个一直在的假设逼到了台面上——因为一旦我们开始让机器自动执行，"谁能命令 UTA"就从理论问题变成了必须回答的问题。

## 5. 怎么修（选项 + 我的推荐，请在这里批注）

### 5.1 推荐方案：给 UTA 交易接口加内部 token 认证

完全复用我们 **#32 已经做过并验证过的模式**（当时给事件接入口 `/api/events/ingest` 加的就是这个）：

1. Guardian（总管进程）每次启动生成一个随机内部 token；
2. Alice 拿到这个 token，每次转发给 UTA 时带上；
3. UTA 拒绝任何**不带正确 token** 的交易请求；
4. 这个 token **从工作区 agent 的环境里剥掉**（就像 #32 已经把 ingest token 剥掉那样，spawn-env 里加一行）。

效果：
- agent 手里没有 token ⟹ 它 `curl` UTA 的 push/kill-switch 路由会被拒 ⟹ **后门锁上了**；
- P3-4a 的 `effectiveAuthzLevel` 变可信（只有认证过的 Alice 能设）⟹ **F1 顺带修好**；
- P1/P2 的"人类专属 push"从"障眼法"变成"真强制"。

代价：中等。改 UTA 路由加认证网关 + Alice 代理带 token + Guardian 生成 token，镜像 #32 的接线。是一个独立 issue（[#51](https://github.com/a-green-hand-jack/OpenAlice/issues/51) 已开）+ 一个 PR 的量。

### 5.2 其他选项（更重，列出供对比）

- **B. 网络隔离工作区**：让 agent 的进程连不到 UTA 的 loopback 端口（命名空间/沙箱）。更彻底，但平台相关、实现重、和现在"同主机多进程"的架构冲突大。
- **C. UTA 改用受限权限的 Unix socket**：不开 TCP 端口，改 socket + 文件权限。但 agent 通常和 Alice/UTA 跑在同一个用户下，除非做用户隔离，否则权限挡不住——收益有限。

我的判断：**A 最对**，成本可控，且我们已经有 #32 的成功先例照抄，还顺手把 F1 和 P1/P2 的既有保证一起夯实。

### 5.3 排期问题（这是要你定的第二件事）

- **选项 甲（推荐）**：把 #51 安全硬化插在 P3-4a **之前**——先锁后门，再让 P3-4a 在认证通道上重新落地（它的核心逻辑基本能复用，改动主要在"档位从哪来"）。
